# M5.5 State-Native Protocol Plan

## Status

- Implemented and game-tested on 2026-07-19.
- Active contract: ABI 2, MCBD v4, and fixed `{stateId:u32 LE}` chunk snapshots.
- Checkpoint A passed with 34817 registry states and exact Lua/native mirror parity.
- Checkpoint B passed for generated plans/models/UVs, coordinate-weighted alternatives, dynamic stairs, dynamic connection blocks, state tint payload, piston state culling, and typed runtime model UV transforms.
- Remaining acceptance is final regression/lifecycle coverage; this document does not claim normalized vertex-packet parity.

## 1. Authority and identity

The new Lua renderer treats `chunk.stateSections` as authoritative. `chunk.blocks` and `chunk.orients` are derived compatibility mirrors and may be lossy for states that cannot be represented by the legacy orientation model.

`stateId` is a runtime ordinal assigned by the finalized state registry. It is not a durable cross-version identifier. The same numeric value is valid only when peers share the exact finalized registry ordering and schema identity.

Protocol requirements:

- state `0` is air;
- state IDs are non-negative and bounded by the finalized registry;
- the current network width is 20 bits, but the native snapshot uses `u32 LE` for simple checked parsing and future capacity;
- native never assigns, sorts, or remaps state IDs;
- persisted or cross-version identity remains canonical block-state text, resolved to the local state ID after loading;
- any registry identity mismatch fails closed and leaves the Lua backend active.

The registry and state transport sources are:

- `lua/mc/sh_block_states.lua`
- `lua/mc/state/sh_network_protocol.lua`
- `lua/mc/state/cl_network_transport.lua`
- `lua/mc/state/sv_network_transport.lua`

## 2. Native chunk snapshot v4

The native bridge receives a complete expanded snapshot for one loaded chunk. It does not receive Lua network packets, packed state-section words, block-entity envelopes, or revision transactions.

For the current configured dimensions:

```text
CS = 16
CH = 32
cell count = CS * CS * CH = 8192
cell record = stateId:u32 LE
snapshot bytes = 32768
```

The local index remains:

```text
li = lx + ly * CS + lz * CS * CS
```

The snapshot is exactly `CS*CS*CH*4` bytes. No header, count, compression, palette, pointers, or trailing bytes are accepted in this first native format. The adapter must expand the authoritative two packed `16^3` state sections before calling `ApplyChunk`.

Native validation must occur before decoding or allocation:

- data pointer is non-null;
- supplied Lua string length equals the exact expected byte count;
- `CS`, `CH`, and derived products are already validated by handshake;
- every decoded state ID is within the uploaded definition range;
- malformed input returns false and does not partially mutate the world mirror;
- old mirror data remains intact if the new snapshot is rejected.

Missing Lua state sections represent air and must be emitted as state ID `0`. The adapter should use `MC.MigrateChunkStateStorage` once, then `MC.GetChunkBlockStateByLocalIndex` or an equivalent checked section traversal. It must not enumerate `chunk.blocks` or derive state IDs through `LegacyFromState`.

## 3. MCBD v4 identity header

MCBD v4 keeps the deterministic little-endian, sequential, length-aware, atomic-parse style of MCBD v3 and adds a state table plus compact generated visual extension.

The exact byte-level implementation will be finalized before coding the parser. The required semantic header fields are:

```text
magic = "MCBD"
version = 4
totalBytes
stateCount
maxStateId
renderDefinitionCount
stateEntryCount
catalog aggregate counts
atlas parameters
schemaHash[32]
visualDefinitionHash[32]
reserved fields = zero
```

Hash meanings:

- `schemaHash` identifies the finalized state registry and therefore protects the `stateId` ordinal mapping;
- `visualDefinitionHash` covers the native render inputs, including state-to-definition mapping, generated plans, model groups and weights, geometry, surfaces, pass/culling metadata, and state-dependent liquid/connection/light metadata;
- these hashes are native-definition compatibility identities and are separate from the Lua network protocol's generation/revision fields.

The parser must reject unknown versions, invalid hash lengths, reserved nonzero data, count bombs, duplicate entries, unknown enum values, invalid indices, invalid atlas extents, non-finite floats, out-of-contract coordinates/UVs, and trailing bytes. Parsing must build a temporary immutable definition set and publish it only after exact tail consumption.

## 4. State entries and render definitions

The implemented lookup chain is:

```text
stateId -> StateEntry -> legacy typed emitter or generated visual plan
```

Each state entry stores the legacy projection, state render flags, and optional five-way stair plans. The generated extension keeps the compact plan/group/model/geometry/surface hierarchy; dynamic Connection and Liquid emitters retain priority where Lua derives geometry from live neighbours.

The six M5 emitter families remain available as typed render-definition kinds:

```text
FullCube | Cross | Model | Shape | Connection | Liquid
```

Legacy custom definitions may still use the six emitter algorithms, but their lookup key is now the resolved state definition rather than a native `blockId/orient` pair.

## 5. Generated visual catalog

The current client catalog is state-indexed and contains shared plan, group, model, geometry, and surface records. Relevant runtime accessors include:

- `MC.StateVisualPlanId`
- `MC.StateVisualPlanRangeFast`
- `MC.StateVisualPlanModelAtFast`
- `MC.StateVisualModelData`

The generated catalog metadata currently reports 32366 states, 12127 visual plans, 6812 model instances, 934 geometry templates, 1843 surface sets, and 681690 packed bytes. These are observed current values, not hard-coded protocol limits.

Generated visual selection can be coordinate-dependent. The native representation must preserve plan/group/weighted alternative information and perform selection using the same integer coordinate hash and weight semantics as Lua. It must not collapse a coordinate-dependent plan into one adapter-time model.

The native protocol must represent data-only records for:

```text
state plan
plan groups
weighted model alternatives
model -> geometry/surface mapping
packed geometry
surface/tile references
```

No Lua function, resolver closure, string evaluation, or executable payload may enter the blob.

## 6. Custom and position-dependent visuals

`MC.ResolveBlockStateVisual(stateId, bx, by, bz)` may combine legacy visual data, generated catalog data, custom state resolvers, and position-dependent values.

The adapter may serialize a visual only when its result is deterministic from typed state/catalog data. A resolver that is arbitrary and position-dependent must either be represented by a future explicitly typed data contract or fail the complete native startup. It must not be sampled at one arbitrary coordinate or silently replaced with the legacy `renderOrient` projection.

The full Lua selection order remains authoritative:

```text
ResolveBlockStateVisual
    -> generated visual plan when generatedVisualOverride
    -> otherwise typed legacy emitter definition
```

State-specific pass, transparency, full-cube, culling, light, liquid, and connection behavior must be captured per state or through a demonstrably compatible shared definition.

## 7. What remains from M3-M5

The following implementation is intentionally outside the protocol redesign and should be preserved:

- one worker and maximum 16 in-flight jobs;
- immutable section snapshots and halo-two capture;
- radius-two dirty influence;
- generation/stale-result rejection;
- opaque/translucent dual pass;
- multi-batch vertex rollover;
- main-thread IMesh creation/destruction;
- candidate staging and atomic section replacement;
- result byte reservations and release on every discard/unload/stop path;
- global Lua fallback on native fault;
- definition seal after the first accepted chunk and reinitialization after clear/shutdown.

## 8. Migration order

1. Add and test state snapshot decoding and state mirror structures.
2. Freeze the complete MCBD v4 byte layout and hostile parser limits.
3. Add Lua state-definition serialization from the finalized registry/catalog.
4. Update emitter lookup and generated-plan selection.
5. Switch the bridge to authoritative state snapshots.
6. Add state-aware diagnostics and parity tests.
7. Run offline tests and compile before game-side visual/lifecycle acceptance.

All migration steps above are implemented. MCBD v3 remains only as the historical M5 contract; ABI 2 rejects old bridge/DLL pairings rather than mixing formats.
